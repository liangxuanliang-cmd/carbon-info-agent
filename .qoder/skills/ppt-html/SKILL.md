# PPT_HTML

创建零依赖、动画丰富、可编辑、完全在浏览器中运行的 HTML 演示文稿。

## 核心原则

1. **零依赖** — 单个 HTML 文件，内联 CSS/JS。无需 npm，无需构建工具。
2. **展示而非描述** — 生成可视化预览，而非抽象选项。人们通过看到效果来发现自己想要什么。
3. **独特设计** — 杜绝千篇一律的"AI 模板感"。每份演示文稿都应有定制感。
4. **视口适配（不可妥协）** — 每张幻灯片必须精确适配 100vh。幻灯片内绝不允许滚动。内容溢出？拆分为多张幻灯片。

## 设计美学

你倾向于生成趋同的、"标准分布"式的输出。在前端设计中，这会产生用户所说的"AI 模板"美学。务必避免：要创造有新意、有特色的前端设计，给用户带来惊喜和愉悦。

重点关注：
- 字体排印：选择美观、独特、有趣的字体。避免 Arial、Inter 等通用字体；选择能提升前端美感的独特字体。
- 色彩与主题：坚持统一的美学风格。使用 CSS 变量保持一致性。以主色搭配鲜明强调色的效果优于平均分配的保守配色。可从 IDE 主题和文化美学中寻找灵感。
- 动效：使用动画实现效果和微交互。HTML 优先使用纯 CSS 方案。React 可用 Motion 库。聚焦高影响力时刻：一个编排精良的页面加载配合交错显示（animation-delay）比分散的微交互更能带来愉悦感。
- 背景：营造氛围和层次感，而非默认使用纯色。叠加 CSS 渐变、使用几何图案或添加与整体美学匹配的情境效果。

避免千篇一律的 AI 生成美学：
- 滥用的字体族（Inter、Roboto、Arial、系统字体）
- 老套的配色方案（尤其是白底紫色渐变）
- 可预测的布局和组件模式
- 缺乏场景特色的模板化设计

发挥创意，做出令人意想不到的选择，让设计真正契合具体场景。在明暗主题、不同字体、不同美学风格之间灵活变换。你仍然倾向于在不同生成中趋同（比如 Space Grotesk）。务必避免：跳出固有思维至关重要！

## 视口适配规则

以下不变量适用于每份演示文稿的每张幻灯片：

- 每个 `.slide` 必须设置 `height: 100vh; height: 100dvh; overflow: hidden;`
- 所有字号和间距必须使用 `clamp(min, preferred, max)` — 绝不使用固定 px/rem
- 内容容器需要 `max-height` 约束
- 图片：`max-height: min(50vh, 400px)`
- 需要高度断点：700px、600px、500px
- 包含 `prefers-reduced-motion` 支持
- 不要直接取反 CSS 函数（`-clamp()`、`-min()`、`-max()` 会被静默忽略）— 使用 `calc(-1 * clamp(...))` 代替

**生成时，请读取 `viewport-base.css` 并将其完整内容包含在每份演示文稿中。**

### 每张幻灯片的内容密度限制

| 幻灯片类型 | 最大内容量 |
|------------|-----------|
| 标题幻灯片 | 1 个标题 + 1 个副标题 + 可选标语 |
| 内容幻灯片 | 1 个标题 + 4-6 个要点 或 1 个标题 + 2 段文字 |
| 特性网格 | 1 个标题 + 最多 6 张卡片（2x3 或 3x2） |
| 代码幻灯片 | 1 个标题 + 8-10 行代码 |
| 引言幻灯片 | 1 段引言（最多 3 行）+ 出处 |
| 图片幻灯片 | 1 个标题 + 1 张图片（最大高度 60vh） |

**内容超出限制？拆分为多张幻灯片。绝不堆砌，绝不滚动。**

## 阶段 0：检测模式

判断用户的需求：

- **模式 A：新建演示文稿** — 从零创建。进入阶段 1。
- **模式 B：PPT 转换** — 转换 .pptx 文件。进入阶段 4。
- **模式 C：增强优化** — 改进现有 HTML 演示文稿。阅读、理解、增强。**遵循下方模式 C 的修改规则。**

### 模式 C：修改规则

增强现有演示文稿时，视口适配是最大风险：

1. **添加内容前：** 统计现有元素数量，对照密度限制检查
2. **添加图片：** 必须设置 `max-height: min(50vh, 400px)`。如果幻灯片已达内容上限，拆分为两张幻灯片
3. **添加文本：** 每张幻灯片最多 4-6 个要点。超出限制？拆分为续页
4. **任何修改后，验证：** `.slide` 有 `overflow: hidden`，新元素使用 `clamp()`，图片有视口相对的 max-height，内容在 1280x720 分辨率下能完整显示
5. **主动重排：** 如果修改会导致溢出，自动拆分内容并通知用户。无需等待用户提出

**向现有幻灯片添加图片时：** 先将图片移至新幻灯片或减少其他内容。绝不在未检查现有内容是否已填满视口的情况下添加图片。

## 阶段 1：内容收集（新建演示文稿）

**在一次 AskUserQuestion 调用中提出所有问题**，让用户一次性填写：

**问题 1 — 用途** （header: "用途"）：
这份演示文稿的用途是什么？选项：路演/融资 / 教学/教程 / 会议演讲 / 内部汇报

**问题 2 — 长度** （header: "长度"）：
大约需要多少张幻灯片？选项：简短 5-10 张 / 中等 10-20 张 / 较长 20+ 张

**问题 3 — 内容** （header: "内容"）：
内容准备情况？选项：内容已齐全 / 有粗略笔记 / 只有主题

**问题 4 — 在线编辑** （header: "编辑"）：
生成后是否需要直接在浏览器中编辑文本？选项：
- "是（推荐）" — 可在浏览器中编辑文本，自动保存到 localStorage，导出文件
- "否" — 仅演示，文件更小

**记住用户的编辑选择 — 这决定了阶段 3 中是否包含编辑相关代码。**

如果用户有内容，请他们分享。

### 步骤 1.2：图片评估（如果提供了图片）

如果用户选择"无图片" → 跳至阶段 2。

如果用户提供了图片文件夹：
1. **扫描** — 列出所有图片文件（.png、.jpg、.svg、.webp 等）
2. **查看每张图片** — 使用 Read 工具（Claude 支持多模态）
3. **评估** — 对每张图片评估：展示了什么、可用或不可用（附原因）、代表什么概念、主色调
4. **协同设计大纲** — 筛选后的图片与文本共同决定幻灯片结构。这不是"先规划幻灯片再添加图片" — 而是从一开始就围绕两者设计（例如：3 张截图 → 3 张功能幻灯片，1 个 logo → 标题/结束幻灯片）
5. **通过 AskUserQuestion 确认** （header: "大纲"）："这个幻灯片大纲和图片选择是否合适？" 选项：看起来不错 / 调整图片 / 调整大纲

**预览中的 Logo：** 如果识别到可用的 logo，将其嵌入（base64）到阶段 2 的每个风格预览中 — 让用户看到自己的品牌以三种不同方式呈现。

---

## 阶段 2：风格探索

**这是"展示而非描述"的阶段。** 大多数人无法用语言表达设计偏好。

### 步骤 2.0：风格路径

询问用户如何选择（header: "风格"）：
- "给我看选项"（推荐）— 根据氛围生成 3 个预览
- "我知道我要什么" — 直接从预设列表中选择

**如果直接选择：** 显示预设选择器并跳至阶段 3。可用预设定义在 [STYLE_PRESETS.md](STYLE_PRESETS.md) 中。

### 步骤 2.1：氛围选择（引导式探索）

询问（header: "氛围"，multiSelect: true，最多选 2 个）：
你希望观众有什么感受？选项：
- 印象深刻/充满信心 — 专业、值得信赖
- 兴奋/充满能量 — 创新、大胆
- 平静/专注 — 清晰、深思熟虑
- 受到启发/感动 — 情感共鸣、令人难忘

### 步骤 2.2：生成 3 个风格预览

根据氛围，生成 3 个不同的单页 HTML 预览，展示字体排印、色彩、动画和整体美学。阅读 [STYLE_PRESETS.md](STYLE_PRESETS.md) 了解可用预设及其规格。

| 氛围 | 建议预设 |
|------|---------|
| 印象深刻/充满信心 | Bold Signal、Electric Studio、Dark Botanical |
| 兴奋/充满能量 | Creative Voltage、Neon Cyber、Split Pastel |
| 平静/专注 | Notebook Tabs、Paper & Ink、Swiss Modern |
| 受到启发/感动 | Dark Botanical、Vintage Editorial、Pastel Geometry |

将预览保存到 `.claude-design/slide-previews/`（style-a.html、style-b.html、style-c.html）。每个预览应自包含，约 50-100 行，展示一张带动画的标题幻灯片。

自动为用户打开每个预览。

### 步骤 2.3：用户选择

询问（header: "风格"）：
你喜欢哪个风格预览？选项：风格 A：[名称] / 风格 B：[名称] / 风格 C：[名称] / 混合元素

如果选择"混合元素"，询问具体细节。

---

## 阶段 3：生成演示文稿

使用阶段 1 的内容（文本，或文本 + 筛选后的图片）和阶段 2 的风格生成完整演示文稿。

如果提供了图片，幻灯片大纲已在步骤 1.2 中整合了图片。如果没有，CSS 生成的视觉效果（渐变、形状、图案）提供视觉趣味 — 这是完全受支持的一等路径。

**生成前，请阅读以下支持文件：**
- [html-template.md](html-template.md) — HTML 架构和 JS 功能
- [viewport-base.css](viewport-base.css) — 必需的 CSS（完整包含）
- [animation-patterns.md](animation-patterns.md) — 所选氛围对应的动画参考

**关键要求：**
- 单个自包含 HTML 文件，所有 CSS/JS 内联
- 在 `<style>` 块中包含 viewport-base.css 的完整内容
- 使用 Fontshare 或 Google Fonts 的字体 — 绝不使用系统字体
- 添加详细注释说明每个部分
- 每个部分都需要清晰的 `/* === 部分名称 === */` 注释块

---

## 阶段 4：PPT 转换

转换 PowerPoint 文件时：

1. **提取内容** — 运行 `python scripts/extract-pptx.py <input.pptx> <output_dir>`（如需安装：`pip install python-pptx`）
2. **与用户确认** — 展示提取的幻灯片标题、内容摘要和图片数量
3. **风格选择** — 进入阶段 2 进行风格探索
4. **生成 HTML** — 转换为所选风格，保留所有文本、图片（来自 assets/）、幻灯片顺序和演讲者备注（作为 HTML 注释）

---

## 阶段 5：交付

1. **清理** — 删除 `.claude-design/slide-previews/`（如存在）
2. **打开** — 使用 `open [filename].html` 在浏览器中启动
3. **询问 PPT 转换** — 询问用户（header: "PPT"）：
   - "需要生成 PPT 文件吗？" 选项："是，生成 PPT" / "不，只需要 HTML"
4. **如果用户需要 PPT** — 将生成的 HTML 转换为 PPTX：
   - 使用 pptxgenjs 创建专业 PPT
   - 匹配所选风格的配色方案和布局
   - 将 `.pptx` 保存到与 HTML 文件相同的目录
   - 创建后打开 PPT 文件
5. **总结** — 告知用户：
   - 文件位置、风格名称、幻灯片数量
   - 导航方式：方向键、空格键、滚动/滑动、点击导航点
   - 如何自定义：`:root` CSS 变量调整颜色，字体链接调整字体排印，`.reveal` 类调整动画
   - 如果启用了在线编辑：鼠标悬停左上角或按 E 进入编辑模式，点击任意文本编辑，Ctrl+S 保存
